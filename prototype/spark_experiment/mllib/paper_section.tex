\subsection{Cold-Path Retraining Benchmark}
\label{sec:retraining}

While the hot path (gateway $\rightarrow$ detector $\rightarrow$ webhook)
must respond to each flow within milliseconds, the \emph{cold path}
of NIDSaaS---periodic retraining of the Tier-2 detection model on
accumulated tenant flow histories---operates under fundamentally
different constraints. Retraining datasets grow with both tenant count
and observation window and rapidly reach the GB-scale at which a
single-node Python ML workflow can no longer load, transform, or fit
the data in memory. This section quantifies that limit empirically
and motivates the inclusion of Apache Spark MLlib in the proposed
architecture.

\subsubsection{Setup}
\label{sec:retraining-setup}

All measurements are taken on a single 12-core / 16~GB / 548~GB Linux
server (Ubuntu~22.04, kernel~5.15) under Docker~28.2. We compare two
RandomForest-classifier implementations using identical hyperparameters
($n\_estimators=100$, $max\_depth=15$, $seed=42$):

\begin{itemize}
\item \textbf{scikit-learn} (Python~3.10): the reference single-node,
      in-memory baseline. Trained directly inside our Python virtual
      environment.
\item \textbf{Apache Spark MLlib} (Spark~3.5.4 via the
      \texttt{apache/spark:3.5.4-python3} container): the distributed
      counterpart we are evaluating. Run in
      \texttt{--master local[*]} mode with a 12~GB driver heap,
      $\textit{spark.memory.fraction}=0.8$, and
      \texttt{MEMORY\_AND\_DISK} persistence so the JVM spills to disk
      when the cached dataset exceeds the heap.
\end{itemize}

The retraining dataset is generated synthetically by a custom
generator we provide alongside the source release. It produces
parquet files (snappy compression) carrying 78 numeric flow features
calibrated against published CIC-IDS2017 statistics, plus a binary
label column at a fixed 80\,\%~benign / 20\,\%~attack split. A fixed
RNG seed makes the entire benchmark deterministic. We sweep three
on-disk dataset sizes---0.1, 0.25, and 0.5~GB---corresponding to
0.84, 2.10, and 4.19 million flow records respectively. Each
\textit{(engine, size)} pair is trained once on an 80/20 stratified
train/test split.

\subsubsection{Discussion}
\label{sec:retraining-discussion}

Three observations frame the architectural decision.

\paragraph{Crossover at $\approx$0.18~GB} sklearn's in-memory tree
induction is optimised for small data: at 0.1~GB it beats Spark
by 1.33$\times$ because Spark's JVM warm-up, broadcast variables,
and shuffle-coordination overhead dominate at low row counts. The
two engines cross above 0.18~GB; from 0.25~GB onward Spark MLlib
holds a 1.5--1.7$\times$ advantage that grows monotonically with
size. This rules out a one-size-fits-all choice: the right engine
depends on where the deployment sits on the size axis.

\paragraph{Single-node ceiling at 1~GB} sklearn at 0.5~GB already
consumes 13.3~GB of resident memory---over 80\,\% of the host's
physical RAM---because pandas, numpy, and \texttt{train\_test\_split}
each maintain their own copies of the feature matrix in float32.
At 1~GB the kernel OOM-kills the process during the
\texttt{pandas.read\_parquet()} call before training even begins.
Spark MLlib hits its own heap limit at 1~GB on this hardware (the
JVM cannot allocate the columnar buffers Spark builds for cached
training partitions), confirming that \emph{single-node} deployments
of either engine are bounded by physical RAM.

\paragraph{Why Spark in the architecture} The above ceiling is the
direct motivation for including Apache Spark in the proposed
architecture (Figure~\ref{fig:arch}). For NIDSaaS deployments
operating over weeks of accumulated multi-tenant flow data---an
inherently multi-GB to multi-TB workload---a single-node Python
retraining path is unworkable: it fails not for lack of CPU but for
lack of RAM. Spark MLlib provides two complementary advantages on
this axis: (i) on a single node it is already
1.5--1.7$\times$ faster than sklearn at the largest sizes both
engines can complete, by virtue of more memory-efficient columnar
caching and parallel split finding; and (ii) the same code path
extends \emph{horizontally} to multiple executor nodes via Spark's
cluster mode, which lifts the single-node memory ceiling
demonstrated above and is the documented production scaling path.
Single-node Python ML libraries do not provide an equivalent
distributed-execution mode and would require a wholesale framework
rewrite to scale beyond one machine. We therefore adopt Spark MLlib
for the cold-path retraining stage, with the understanding that
production deployments will operate it in cluster mode rather than
the single-node configuration we benchmark here.

\paragraph{Threat to validity}\label{sec:retraining-threats}
Our dataset is synthetic (advisor-approved), with feature
distributions calibrated against published CIC-IDS2017 statistics
but not drawn from real packet traces. The classes are clean enough
that both engines achieve $\geq$0.9999 F1 on every run, so the
benchmark isolates \emph{scaling behaviour}, not classification
quality. Real-world data with overlapping benign/attack distributions
would lengthen training (more node splits before purity termination)
but would not change the qualitative scaling story---Spark's
distributed split-finding remains the only architectural path past
the 1-GB single-node ceiling we observed. Multi-node Spark
performance is left to production deployment as future work.
